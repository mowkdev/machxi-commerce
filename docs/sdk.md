# Admin SDK

The admin frontend talks to the API exclusively through `@repo/admin-sdk`, a
TypeScript client generated from an OpenAPI spec the API server emits.

## Pipeline

```
apps/api/src/<feature>/routes.ts        zod request/response schemas registered
       │                                via hono-openapi `describeRoute`
       ▼
apps/api/openapi.json                   emitted by `pnpm openapi:emit`
       │                                (loads createApp(), generateSpecs)
       ▼
packages/admin-sdk/src/gen/             generated by `pnpm sdk:generate`
  ├── client/                           per-route axios call functions
  ├── hooks/                            TanStack Query hooks (use*, use*Suspense)
  ├── types/                            response/request/error types
  └── zod/                              client-side request validators
       │
       ▼
apps/admin/src/features/<feature>/      thin app-side hooks compose SDK hooks
                                        with toast, navigation, invalidation
```

The SDK is the API layer. There is no per-feature `api.ts` indirection.

## Regenerating after API changes

After any change to a route's schemas:

```bash
pnpm sdk:generate
```

This runs two steps:

1. `pnpm --filter @app/api openapi:emit` — boots `createApp()`, calls
   `generateSpecs`, writes `apps/api/openapi.json`. Does **not** require the
   database or S3 to be running.
2. `pnpm --filter @repo/admin-sdk generate` — kubb reads the spec and
   regenerates `packages/admin-sdk/src/gen/`.

Both `openapi.json` and the `gen/` output are committed. CI verifies they're
up to date by re-running `pnpm sdk:generate` and failing on a non-empty diff.

## Response envelope

Every API response is wrapped:

```ts
// success
{ success: true, data: T, meta?: PaginationMeta }
// error
{ success: false, error: { code: string, message: string, details?: ... } }
```

The OpenAPI spec models the full envelope. SDK types and runtime behavior
both reflect this:

- Generated functions return `{ success: true, data, meta? }` on success.
- The runtime throws `SdkRequestError` on non-2xx (mapped from the error
  envelope: `code`, `message`, `status`, `details`).

Feature-level hooks unwrap with React Query's `select`:

```ts
useAdminGetCategory<CategoryDetail>(id, {
  query: { select: (response) => response.data },
});
```

## Cache key convention

Every feature exports a query prefix that matches the SDK's URL key:

```ts
// e.g. apps/admin/src/features/categories/hooks.ts
export const categoriesQueryPrefix = [{ url: '/api/categories' }] as const;
```

- `AppDataGrid`'s `queryKey` prop is set to this prefix, so its internal
  `useQuery` shares a key namespace with SDK hooks.
- Mutations call
  `queryClient.invalidateQueries({ queryKey: <prefix> })` once — that
  catches list views, detail queries, and option pickers in one shot.

## Adding a new route

1. Add the request/response zod schemas to `@repo/types/admin` (or feature
   schema if specific to the API).
2. In `apps/api/src/<feature>/routes.ts`, register the route with
   `describeRoute({ operationId, summary, tags, requestBody, parameters,
   responses })`. Use the helpers in
   `apps/api/src/openapi/envelope.ts`:
   - `successEnvelope(T)` / `paginatedEnvelope(item)`
   - `jsonRequestBody(schema)`
   - `jsonResponse(description, schema)`
   - `paramsFromSchema(zodObject, 'query' | 'path')`
   - `standardErrorResponses` (spread to add 4xx/5xx descriptions)
3. Run `pnpm sdk:generate`.
4. In `apps/admin/src/features/<feature>/hooks.ts`, add a thin hook that
   calls the generated SDK hook (`useAdminFoo`) or function (`adminFoo`).

## Multipart uploads

Multipart endpoints (e.g. media upload) are declared in OpenAPI as
`multipart/form-data` with a permissive schema. The SDK runtime detects
`FormData` payloads and strips the default `Content-Type` header so axios
sets its own boundary.

## Known constraints

- **Schemas are inlined in the spec, not hoisted to `components/schemas`.**
  zod v4 + `.meta({ id })` currently emits `$ref` paths that don't resolve
  through `@standard-community/standard-openapi`'s zod adapter. Letting
  schemas inline is verbose but correct. Revisit when upstream fixes
  hoisting.
- **Forms are not generated.** Form schemas in
  `apps/admin/src/features/<feature>/schema.ts` exist for UX
  (flat field shapes, custom error messages). They are deliberately not the
  same shape as the SDK request DTOs. Don't try to unify them.
- **Generated files use `.ts` extensions in import paths.** The
  `@repo/admin-sdk` tsconfig sets `allowImportingTsExtensions: true` and
  consumes the package source directly (no build step).

## Coexistence with the legacy axios client

`apps/admin/src/lib/api.ts` is the original axios instance with an
envelope-unwrap interceptor. It is now used only by code that hasn't been
migrated:

- Auth login / session checks
- Anything outside the four catalog features

When all surfaces are on the SDK, that file can be retired.
